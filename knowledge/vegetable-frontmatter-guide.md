# 野菜データベース frontmatter 設計ガイド

このガイドは、野菜データベースを Skills で新規作成するときに、AI が frontmatter を自動入力しやすくし、人間が後から修正しやすくするための項目説明です。

基本方針:

- 迷った項目は空欄にするか、候補を控えめに入れる。
- 小川農園固有の情報は、根拠がある場合だけ入れる。
- 一般情報を入れる場合は、あとで確認できるように本文の参考情報源にも出典を残す。
- 値は検索・分類・絞り込みに使いやすい粒度にする。
- 文章ではなく、短い分類語・タグ・用途語を中心にする。

## 推奨 frontmatter


## category_large

野菜を大きく分類するための項目です。売り場・一覧・検索で大きなまとまりを作る粒度にします。

入力の目安:

- 1つだけ入れる。
- 食べ方ではなく、野菜としての大分類を入れる。
- 判断に迷う場合は一般的な流通分類を優先する。

参考例:

- 果菜類
- 葉菜類
- 根菜類
- いも類
- 豆類
- 香味野菜
- ハーブ類
- 山菜・野草
- 西洋野菜
- 穀物・雑穀
- きのこ類

例:

```yaml
category_large: 果菜類
```

## category_medium

`category_large` の下に置く中分類です。品目の特徴や売り場での探しやすさを補います。

入力の目安:

- 1つだけ入れる。
- 大分類より具体的に、品目グループが分かる粒度にする。
- 品種名ほど細かくしない。

参考例:

- トマト類
- ナス類
- ウリ類
- ピーマン・唐辛子類
- 葉物
- 結球野菜
- 根もの
- かぶ・大根類
- いも類
- 豆・さや豆類
- 香味・薬味
- ハーブ
- 西洋葉菜
- 茎を食べる野菜

例:

```yaml
category_medium: トマト類
```

## category_small

品目名や用途上の細分類です。ユーザーが「何の野菜か」を見て分かる粒度にします。

既存データベースの重複判定に使う唯一の品目IDです。旧 `vegetable` frontmatter は使わず、`category_small` を基準にします。

入力の目安:

- 基本は野菜名を入れる。
- ミニトマト、加賀太きゅうり、ビーツなど、商品として区別したい単位にする。
- 品種名が販売上重要な場合だけ品種名まで入れる。

参考例:

- トマト
- ミニトマト
- ナス
- 水ナス
- キュウリ
- ジャガイモ
- ルバーブ
- アーティチョーク
- ビーツ
- ケール
- バジル
- スナップエンドウ

例:

```yaml
category_small: ミニトマト
```

## crop_family

植物分類上の科を入れる項目です。栽培管理や輪作、アレルギー・苦手傾向の把握にも使えます。

入力の目安:

- `ナス科` のように「科」まで入れる。
- 不明な場合は空欄にする。
- AIが一般情報で入れる場合は、Webなどで確認したものを使う。

参考例:

- ナス科
- ウリ科
- アブラナ科
- キク科
- ヒユ科
- セリ科
- ユリ科
- ヒガンバナ科
- マメ科
- イネ科
- タデ科
- シソ科

例:

```yaml
crop_family: ナス科
```

## edible_part

主に食べる部位を入れる項目です。調理方法や紹介文の方向性を決めやすくします。

入力の目安:

- 複数ある場合は配列にする。
- 一般的に食べる部位を入れる。
- ルバーブの葉のように食用にしない部位は、本文の注意点に書く。

参考例:

- 果実
- 葉
- 茎
- 根
- 塊茎
- 鱗茎
- 花蕾
- さや
- 種子
- 新芽
- 香り葉

例:

```yaml
edible_part: 果実
```

複数例:

```yaml
edible_part:
  - 葉
  - 茎
```

## color

見た目の主な色を入れる項目です。売り場の彩り、セット提案、SNS紹介に使います。

入力の目安:

- 基本は主色を1つ入れる。
- 色違いセットやカラフル品種の場合は配列にする。
- 色味は細かすぎず、人が見て分かる表現にする。

参考例:

- 赤
- 黄
- オレンジ
- 緑
- 濃緑
- 紫
- 白
- 黒紫
- ピンク
- クリーム
- 複色
- カラフル

例:

```yaml
color: 赤
```

複数例:

```yaml
color:
  - 赤
  - 黄
  - 緑
```

## shape

形状の特徴を入れる項目です。見た目の説明や、商品選びの手がかりになります。

入力の目安:

- 代表的な形を短い言葉で入れる。
- 品種によって大きく変わる場合は、本文で補足する。
- 「かわいい」「きれい」など感想ではなく形状語にする。

参考例:

- 丸い
- 楕円
- 細長い
- 太長い
- 平たい
- 球形
- 円錐形
- 房状
- さや状
- 葉状
- 茎状
- ごつごつ
- しずく形

例:

```yaml
shape: 丸い
```

## size_class

サイズ感を分類する項目です。規格・詰め合わせ・調理用途の目安にします。

入力の目安:

- 厳密な重量ではなく、販売や説明に使いやすい大まかな分類にする。
- 必要なら本文に実寸や重量を書く。
- 品種や収穫時期で変わる場合は代表的なサイズを入れる。

参考例:

- 極小
- 小
- 中
- 大
- 特大
- ミニ
- 一口サイズ
- 細め
- 太め
- 長め
- 規格外あり

例:

```yaml
size_class: 中
```

## texture

入力の目安:

- 1〜3語程度の短い表現にする。
- 生と加熱で変わる場合は代表値を入れ、本文で補足する。
- 迷う場合は空欄にする。

参考例:

- みずみずしい
- しゃきしゃき
- ぱりっと
- ほくほく
- ねっとり
- とろり
- やわらかい
- 歯切れがよい
- 肉厚
- なめらか
- 繊維感あり
- ほろ苦い食感

例:

```yaml
texture: みずみずしい
```

## flavor

味わいの方向性を入れる項目です。POPや紹介文の「味の軸」にします。

入力の目安:

- 1〜3語程度の短い表現にする。
- 甘味・酸味・苦味・香りなど、食べたときの印象を入れる。
- 栽培条件や個体差で強く変わる表現は控えめにする。

参考例:

- 甘い
- 酸味あり
- 甘酸っぱい
- ほろ苦い
- 香りがよい
- さっぱり
- 濃い味
- クセが少ない
- うまみがある
- 辛みあり
- 青みがある
- 土の香り

例:

```yaml

```

- 調理法、使いどころ、買う人の目的が混ざっていてもよい。

## sales_angle

販売・紹介で打ち出しやすい切り口を配列で入れる項目です。POP、SNS、セット名、商品説明の方向性に使います。

入力の目安:

- 複数選択可。
- 事実として言えるものだけ入れる。
- 「珍しい」「旬」などは、販売時期や地域感で変わるため必要に応じて見直す。
- 小川農園固有の強みは、根拠がある場合だけ入れる。

## AI入力時の判断ルール

AIが自動入力する場合は、次の順で判断します。

1. 既存の小川農園データベースに書かれている事実を優先する。
2. ユーザーが今回入力した情報を反映する。
3. 一般情報入りテンプレートの場合は、Webで確認した一般情報を入れる。
4. 判断できない項目は空欄にする。
5. 販売上の強い表現は、根拠がなければ入れない。

避けたい入力:

- `おいしい`
- `最高`
- `絶品`
- `安心安全`
- `栄養満点`
- `健康にいい`
- `誰でも好き`

理由:

- 主観的・誇張的な表現は、データベースの分類値として使いにくい。
- 健康・栄養の断定は根拠確認が必要。
- 紹介文では使える場合があっても、frontmatter では検索しやすい客観語を優先する。

## 人間が修正するときの見方

修正時は、次の観点で見直します。

- 分類が細かすぎないか
- 販売上の表現が事実に基づいているか
- 味・食感が小川農園の実際の商品と合っているか
- 色やサイズが品種違いに対応できているか
- `best_uses` と `sales_angle` が紹介文・POP・SNSで使いやすいか

迷った場合は、frontmatter は控えめにして、詳しい説明を本文に書くのがおすすめです。
